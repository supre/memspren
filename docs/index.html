MemSpren: The Cognitive Journal

A personal knowledge system at the synthesis of psychology and tech. Engineered for clarity, not just storage.

Clarity, Not Motivation

Most productivity tools fail by asking for more friction. MemSpren focuses on reducing the energy barrier to synthesis. We don't gamify your memory; we curate your attention.

The Through-Line

A living document isn't a static archive. It's a continuous thread of thought across days, months, and years. Our architecture ensures that yesterday's insight fuels tomorrow's breakthrough.

Philosophy

This isn't just a storage system. The goal is persistent context without manual effort:

  • check_circle Your AI reads recent notes on every interaction
  • check_circle You never have to re-explain yourself
  • check_circle Your thinking is captured atomically, linked, and queryable
  • check_circle Nothing gets deleted — everything gets archived
  • check_circle Your vault is portable, future-proof, and yours forever

Architecture

Two-Location Design

Every plugin follows this pattern:

Location What lives here
Skill folder config.md, Memory/, operational files
Vault All note content: Work/, People/, Log/, etc.

Entity Types

  • — Projects — deadline-driven work with goals and progress notes
  • — Ideas — atomic thoughts, underdeveloped concepts, seeds for future projects
  • — People — everyone you work with, interaction logs, connections to projects
  • — Tasks — unified inbox with priority and connections to projects/ideas
  • — Learnings — things you've learned, read, understood, connected to active work
  • — Daily logs — one note per check-in, hub of all entities touched that day

Vault Structure

[vault_path]/
├── Vision/          — values, long-term vision, guiding principles
├── Strategy/        — quarterly goals and active strategies
├── Work/
│   ├── Projects/    — deadline-driven work
│   └── Ideas/       — lightweight captures
├── People/          — work contacts and interaction logs
├── Tasks/           — unified task inbox
├── Log/Daily/       — one note per check-in day
├── Notes/Learnings/ — things learned, connected to work
├── Inbox/           — unsorted catch-all
└── Archive/         — nothing ever deleted

Smart Merge Architecture

MemSpren v0.02 — Internal Architecture Document

Date: March 21, 2026

Author: Supreet Singh

Status: Implemented

Affects: OpenClaw skill only

Problem Solved: Duplicate file creation, vault clutter, no version control safety net

Problem Statement

During production use of MemSpren v0.02's batch sync system, three critical quality issues surfaced:

  1. Duplicate documents: Multiple files created for the same topic
  2. No existence checking: Sync process created new files without checking if a similar file already existed
  3. No safety net: When files were modified, the previous version was lost

Solution: Smart Merge + Git Safety

Script 1: scan_descriptions.py NEW

Scans a vault folder and extracts frontmatter metadata from all files. Returns structured JSON so the agent can find similar/duplicate files before creating new ones.

python3 scripts/scan_descriptions.py \
  --vault-path "/path/to/vault" --folder "Vision"

# Output (JSON)
[{
  "path": "Vision/Psychology-Tech Synthesis.md",
  "description": "Career through-line combining psychology and technology",
  "node_type": "vision",
  "status": "active",
  "last_modified": "2026-03-15"
}]

Key design decision: The script only reads frontmatter (first few lines until closing ---), not the full file. This keeps token cost minimal even when scanning folders with many files.

Script 2: git_commit.py NEW

Commits the current state of a file before modification. Creates a safety checkpoint — nothing is ever lost.

python3 scripts/git_commit.py \
  --vault-path "/path/to/vault" \
  --file "Vision/Psychology-Tech Synthesis.md" \
  --message "pre-merge snapshot"

# Output: { "action": "committed", "hash": "abc123f", ... }

# Initialize git in vault (during setup)
python3 scripts/git_commit.py --vault-path "/path/to/vault" --init

The description Field

Every file in the vault now requires a description field in its YAML frontmatter. It serves three purposes:

  • — Merge key: Used by scan_descriptions.py to find similar files
  • — File identity: The agent reads descriptions to decide merge vs create
  • — Retrieval aid: Future "what do I have about X?" queries can search descriptions 
---
node_type: idea
created: 2026-03-21
description: "Gamification of daily habits with points earned for right actions"
status: active
connected: []
tags: []
last_modified: 2026-03-21
---

Merge Scope Rules

Node Type Folder Behavior
visionVision/Aggressive — few files, merge readily
strategyStrategy/Aggressive — few files
ideaWork/Ideas/Moderate — similar ideas consolidate
learningNotes/Learnings/Moderate — same topic = same file
patternNotes/Patterns/Conservative — patterns are distinct
projectWork/Projects/No scan — always distinct
personPeople/No scan — always distinct
log-entryLog/Daily/Date-based — same date = append
ktWork/KT/Moderate — same system = same file

Git as Safety Net

Setup requirement: Git must be installed before MemSpren setup. The setup protocol initializes a git repo in the vault with an initial commit. git_commit.py handles initialization if the vault isn't already a repo.

When new content conflicts with existing content, the agent:

  1. Commits the current file state (git snapshot)
  2. Updates the file with reconciled content
  3. Notifies the user with what changed and git hash for rollback
"I updated [file].
 Previous version said: [X]
 New information says: [Y]
 I've reconciled it to: [Z]

 Previous version saved in git (commit abc123f).
 Run `git diff abc123f` to see what changed.
 Run `git revert abc123f` to undo."

What This Doesn't Solve (Future)

  • — Cross-folder merging — e.g., an idea that should become a project
  • — Semantic similarity — would need embeddings for truly intelligent matching
  • — Retroactive cleanup — existing duplicates need a separate cleanup pass
  • — Auto-adding descriptions — existing files without descriptions need migration
