Problem Solved: Duplicate file creation, vault clutter, no version control safety net

Problem Statement

During production use of MemSpren v0.02's batch sync system, three critical quality issues surfaced:

  1. Duplicate documents: Multiple files created for the same topic (e.g., multiple vision documents, similar idea files)
  2. No existence checking: The sync process created new files without checking if a similar file already existed in the target folder
  3. No safety net: When files were modified, the previous version was lost — no version control, no rollback capability

These issues compounded with the multi-sync cron jobs (noon, 5 PM, 9 PM), where each sync could create duplicate entities independently.

Solution: Description-Based Smart Merge + Git Safety

Two new Python scripts provide governance over all Obsidian writes during sync:

Script 1: scan_descriptions.py NEW

Scans a vault folder and extracts frontmatter metadata from all files. Returns structured JSON so the agent can find similar/duplicate files before creating new ones.

# Usage
python3 scripts/scan_descriptions.py \
  --vault-path "/path/to/vault" \
  --folder "Vision"

# Output (JSON)
[
  {
    "path": "Vision/Psychology-Tech Synthesis.md",
    "description": "Career through-line combining psychology and technology",
    "node_type": "vision",
    "status": "active",
    "last_modified": "2026-03-15"
  }
]
Key design decision: The script only reads frontmatter (first few lines until closing ---), not the full file. This keeps token cost minimal even when scanning folders with many files.

Script 2: git_commit.py NEW

Commits the current state of a file before modification. Creates a safety checkpoint — nothing is ever lost.

# Usage
python3 scripts/git_commit.py \
  --vault-path "/path/to/vault" \
  --file "Vision/Psychology-Tech Synthesis.md" \
  --message "pre-merge snapshot: before appending new vision context"

# Output (JSON)
{
  "action": "committed",
  "hash": "abc123f",
  "file": "Vision/Psychology-Tech Synthesis.md",
  "message": "pre-merge snapshot: ..."
}

# Initialize git in vault (during setup)
python3 scripts/git_commit.py --vault-path "/path/to/vault" --init

Frontmatter: The description Field

Every file in the vault now requires a description field in its YAML frontmatter:

---
node_type: idea
created: 2026-03-21
description: "Gamification of daily habits with points earned for right actions and lost for missed commitments"
status: active
connected: []
tags: []
last_modified: 2026-03-21
---

The description serves three purposes:

  1. Merge key: Used by scan_descriptions.py to find similar files
  2. File identity: The agent reads descriptions to decide merge vs create
  3. Retrieval aid: Future "what do I have about X?" queries can search descriptions

Git as Safety Net

BEFORE MODIFICATION AFTER MODIFICATION │ │ ▼ ▼ ┌─────────────────┐ ┌─────────────────┐ │ git commit │ │ File updated │ │ "pre-merge │ │ with new content│ │ snapshot" │ │ │ │ │ │ If user │ │ hash: abc123f │ ──────► │ disagrees: │ │ │ │ git diff abc123f│ │ NOTHING LOST │ │ git revert │ └─────────────────┘ └─────────────────┘
Setup requirement: Git must be installed before MemSpren setup proceeds. The setup protocol now initializes a git repo in the vault with an initial commit of all existing files. The git_commit.py script handles initialization if the vault isn't already a repo.

Conflict Resolution: User Notification

When new content conflicts with existing content (not just adds to it), the agent:

  1. Commits the current file state (git snapshot)
  2. Updates the file with reconciled content
  3. Sends the user a notification:
"I updated [file].
 Previous version said: [X]
 New information says: [Y]
 I've reconciled it to: [Z]

 Previous version saved in git (commit abc123f).
 Run `git diff abc123f` to see exactly what changed.
 Run `git revert abc123f` to undo if you disagree."

What This Doesn't Solve (Future)

  • Cross-folder merging — e.g., an idea that should become a project (different concern, different workflow)
  • Semantic similarity — would need embeddings for truly intelligent matching (current approach is description-based keyword matching by the agent)
  • Retroactive cleanup — existing duplicate files need a separate cleanup cron job to identify and consolidate
  • Auto-adding descriptions — existing files without descriptions need a migration pass
